# RAG 기반 로컬 문서 탐색기

사용자가 직접 제공한 프로젝트 문서, 보고서, 견적서, 예산 자료, 회의 자료 등을 로컬에서 학습한 뒤 질문으로 검색하고 필요한 양식 파일을 찾는 RAG 기반 문서 탐색 도구입니다.

이 저장소에는 기본 학습 문서가 포함되어 있지 않습니다. 문서와 대화 기록은 실행한 사람의 로컬 PC 안에서만 저장됩니다. GitHub 저장소에는 개인 문서, 원본 데이터, 벡터 DB가 포함되지 않도록 `.gitignore`에 제외해두었습니다.

## 주요 기능

- 웹 화면에서 PDF, TXT, MD, CSV 문서 업로드
- 업로드한 문서 즉시 학습 및 검색
- Ollama 로컬 모델을 이용한 답변 생성
- 참고 문서 다운로드
- 검색 결과에 `이 문서가 맞음` / `이 문서 아님` 피드백 저장
- GPT처럼 대화 기록 저장, 검색, 삭제
- 모든 개인 데이터는 `data/`, `chroma_db/`, `chat_histories/` 아래 로컬 파일로 관리

## 설치 및 실행 가이드

README만 보고 실행할 수 있도록 아래 순서대로 진행합니다.

### 1. Python 준비

Python 3.10 이상을 권장합니다. 설치 확인은 PowerShell에서 아래 명령으로 확인합니다.

```powershell
python --version
```

### 2. Ollama 설치

이 앱은 답변 생성을 위해 로컬 LLM 실행 도구인 Ollama를 사용합니다. Ollama는 사용자의 PC에서 모델을 실행하므로 문서 내용을 외부 API로 보내지 않습니다.

1. Ollama 공식 사이트에서 Windows용 설치 파일을 내려받아 설치합니다.
2. 설치 후 PowerShell에서 아래 명령으로 설치 여부를 확인합니다.

```powershell
ollama --version
```

### 3. 권장 모델 다운로드

아래 모델을 권장합니다.

```powershell
ollama pull qwen2.5:7b-instruct
```

모델이 설치됐는지 확인합니다.

```powershell
ollama list
```

앱 기본 모델명은 `qwen2.5:7b-instruct`입니다. Streamlit 화면의 사이드바에서도 모델명을 바꿀 수 있습니다.

### 4. Python 가상환경 및 패키지 설치

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### 5. 실행 방식 선택

이 프로젝트는 실행 방식이 크게 두 가지입니다.

#### 5-1. Streamlit 앱 실행 방식

일반 사용자가 브라우저에서 문서를 업로드하고 질문하는 실제 화면입니다. 시연이나 일반 사용은 이 방식을 사용합니다.

```powershell
.\venv\Scripts\python.exe -m streamlit run app_streamlit.py --server.port 8501
```

브라우저에서 아래 주소로 접속합니다.

```text
http://localhost:8501
```

#### 5-2. FastAPI 서버 실행 방식

다른 프로그램, 별도 웹 프론트엔드, 자동화 스크립트와 연결할 때 사용하는 API 서버 방식입니다. `main.py`가 `/upload`, `/search`, `/status`, `/learning-status`, `/ingest`, `/reset` 같은 API를 제공합니다.

```powershell
.\venv\Scripts\python.exe main.py
```

서버가 실행되면 기본 주소는 아래와 같습니다.

```text
http://localhost:8000
```

API 문서는 FastAPI가 자동으로 제공하는 Swagger 화면에서 확인할 수 있습니다.

```text
http://localhost:8000/docs
```

정리하면 다음과 같습니다.

| 실행 방식 | 파일 | 주소 | 용도 |
|---|---|---|---|
| Streamlit | `app_streamlit.py` | `http://localhost:8501` | 사용자가 직접 쓰는 웹 화면 |
| FastAPI | `main.py` | `http://localhost:8000` | 외부 프로그램/프론트엔드 연동용 API |

### 6. 실행 전 확인 사항

- Ollama가 설치되어 있어야 합니다.
- `qwen2.5:7b-instruct` 모델이 `ollama pull`로 내려받아져 있어야 합니다.
- 처음 실행할 때 `sentence-transformers` 임베딩 모델을 내려받기 때문에 시간이 걸릴 수 있습니다.
- 앱 주소 `localhost:8501`은 현재 실행 중인 내 PC에서만 접속되는 로컬 주소입니다.

## 사용 방법

1. 화면 상단의 `문서 업로드 및 학습`을 엽니다.
2. 사용자가 직접 준비한 PDF, TXT, MD, CSV 파일을 선택합니다.
3. `업로드한 문서 학습` 버튼을 누릅니다.
4. 채팅창에 필요한 자료나 양식을 질문합니다.
5. 참고 문서에서 다운로드하거나, 결과가 맞으면 `이 문서가 맞음`을 누릅니다.
6. 잘못 나온 문서는 `이 문서 아님`을 눌러 다음 검색 순위에서 낮춥니다.

## 민감정보 마스킹 다운로드

시연이나 공유 상황에서는 문서 안에 이름, 전화번호, 이메일, 학번 같은 개인정보가 포함될 수 있습니다. 이 앱은 다운로드 시 민감정보 패턴을 검사하고, 감지된 경우 원본 대신 마스킹본을 제공합니다.

```text
민감정보 없음 → 원본 파일 다운로드
민감정보 감지 → 개인정보를 삭제/마스킹한 파일 다운로드
```

현재 감지하는 주요 항목은 다음과 같습니다.

- 이메일
- 전화번호
- 주민등록번호 형태
- 학번/식별번호 형태
- `이름`, `성명`, `담당자`, `작성자`, `신청자` 같은 라벨 뒤의 이름

마스킹본 제공 방식은 파일 형식에 따라 다릅니다.

| 파일 형식 | 처리 방식 |
|---|---|
| `.txt`, `.md`, `.csv` | 같은 텍스트 계열 확장자로 마스킹본 제공 |
| `.xlsx` | 엑셀 형식을 유지한 마스킹본 제공 |
| `.docx` | 워드 형식을 유지한 마스킹본 제공 |
| `.pptx` | 파워포인트 형식을 유지한 마스킹본 제공 |
| `.pdf` | 추출 텍스트를 기반으로 새 마스킹 PDF 제공 |
| `.xls`, `.doc`, `.ppt` | 자동 마스킹 저장이 불안정하면 다운로드 제한 |

이 기능은 완전한 개인정보 탐지 시스템이 아니라 시연 안전성을 높이기 위한 보조 장치입니다. 실제 배포 환경에서는 추가 검수와 더 강한 개인정보 탐지 정책이 필요합니다.

## 대화 기록 관리

채팅 기록은 외부 서버가 아니라 로컬 폴더에 JSON 파일로 저장됩니다.

```text
chat_histories/
```

대화 기록은 다음 방식으로 관리됩니다.

- 첫 사용자 질문을 바탕으로 대화 제목을 자동 생성합니다.
- 사이드바에서 이전 대화를 검색할 수 있습니다.
- 사이드바에서 이전 대화를 다시 불러올 수 있습니다.
- 필요 없는 대화는 삭제할 수 있습니다.
- 저장된 대화 기록은 GitHub에 포함하지 않습니다.

즉, 대화 기록도 문서 데이터와 마찬가지로 로컬 PC 안에서만 관리됩니다.

## 새 문서 등록 시 동작 방식

이 앱은 서버형 서비스가 아니라 로컬 PC 안에서만 동작하는 문서 검색 도구입니다. 저장소에는 기본 문서가 들어 있지 않으며, 사용자가 웹 화면에서 직접 제공한 문서만 학습합니다. 새 문서를 업로드해도 외부 서버나 GitHub로 전송되지 않고, 현재 실행 중인 PC의 로컬 폴더와 로컬 DB에만 저장됩니다.

새 문서를 등록하면 아래 순서로 처리됩니다.

```text
1. 사용자가 웹 화면에서 직접 준비한 문서 업로드
2. 업로드 원본 파일이 로컬 원본 폴더에 저장됨
   data/documents/원본데이터/사용자 업로드/
3. PDF/TXT/MD/CSV는 학습용 폴더로 복사됨
   Office/HWP 파일은 가능한 경우 PDF로 변환되어 학습용 폴더에 저장됨
   data/documents/PDF화 된 문서들/사용자 업로드/
4. 학습용 파일에서 텍스트 추출
5. 텍스트가 900자 단위 조각으로 나뉨
6. 각 조각을 임베딩 벡터로 변환
7. ChromaDB 로컬 벡터 DB에 저장
   chroma_db/
8. 이후 질문부터 새 문서도 검색 대상에 포함됨
```

즉, 새 문서를 추가한 뒤 `업로드한 문서 학습` 버튼을 누르면 바로 검색 대상이 됩니다. 검색에는 학습용 PDF/TXT/MD/CSV가 사용되고, 다운로드는 가능한 경우 `원본데이터`에 저장된 원본 파일을 제공합니다. 앱을 껐다 켜도 `chroma_db/` 폴더가 남아 있으면 기존 학습 결과를 다시 사용할 수 있습니다.

### 문서 추가 흐름도

웹 화면에서 새 문서를 추가하면 원본 파일과 검색 학습용 파일을 분리해서 관리합니다.

```text
[사용자 업로드 파일]
       │
       ▼
[원본 파일 저장]
data/documents/원본데이터/사용자 업로드/
       │
       ├─ PDF/TXT/MD/CSV
       │      │
       │      └─ 학습용 폴더로 복사
       │
       └─ XLS/XLSX/DOC/DOCX/PPT/PPTX/HWP/HWPX
              │
              └─ 로컬 Office/HWP 프로그램으로 PDF 변환
                     │
                     ▼
[학습용 파일 저장]
data/documents/PDF화 된 문서들/사용자 업로드/
       │
       ▼
[DocumentRAG.ingest_file()]
       │
       ├─ 텍스트 추출
       ├─ 900자 chunk 분할
       ├─ 임베딩 생성
       └─ ChromaDB 저장
              │
              ▼
[검색 가능 상태]
       │
       ▼
[참고 문서 다운로드]
가능하면 원본데이터에 저장된 원본 파일을 제공
```

### 로컬 DB의 역할

이 프로젝트에서 말하는 DB는 외부 데이터베이스가 아니라 프로젝트 폴더 안에 생성되는 로컬 ChromaDB입니다.

```text
chroma_db/
```

이 폴더에는 문서 원문 전체가 아니라, 검색을 위한 문서 조각 정보와 임베딩 벡터가 저장됩니다. 사용자가 질문하면 앱은 이 로컬 DB에서 질문과 의미가 가까운 문서 조각을 찾습니다.

### 로컬 저장 구조

현재 로컬에서 실행할 때 기준 구조는 아래와 같습니다.

```text
AI project/
├─ data/
│  └─ documents/
│     ├─ 원본데이터/
│     │  └─ 사용자 업로드/
│     │     └─ 사용자가 업로드한 원본 문서
│     │
│     └─ PDF화 된 문서들/
│        └─ 사용자 업로드/
│           └─ 검색 학습용 PDF/TXT/MD/CSV 문서
│
├─ chroma_db/
│  └─ ChromaDB 로컬 벡터 DB
│     └─ 검색용 임베딩 벡터와 문서 조각 메타데이터
│
├─ chat_histories/
│  └─ 로컬 대화 기록
│
└─ data/
   └─ feedback/
      └─ 검색 결과 피드백
```

정리하면 원본 파일은 `data/documents/원본데이터/...` 아래에 저장되고, 검색 학습용 파일은 `data/documents/PDF화 된 문서들/...` 아래에 저장됩니다. 검색을 빠르게 하기 위한 벡터 DB는 `chroma_db/`에 저장됩니다. 모두 로컬 PC 안의 파일/폴더이며 외부 DB 서버를 사용하지 않습니다.

### 로컬에 저장되는 정보

앱 사용 중 생성되는 주요 로컬 데이터는 다음과 같습니다.

```text
data/documents/원본데이터/...        업로드한 원본 문서 파일
data/documents/PDF화 된 문서들/...   검색 학습용 파일
chroma_db/            문서 검색용 벡터 DB
chat_histories/       대화 기록
data/feedback/        검색 결과 피드백
```

이 폴더들은 개인 문서나 사용 기록을 포함할 수 있으므로 GitHub에 올리지 않도록 `.gitignore`에 제외되어 있습니다.

## RAG 검색 구조

이 앱의 검색은 문서를 통째로 비교하지 않고, 문서를 작은 조각으로 나눈 뒤 질문과 가까운 조각을 찾는 방식입니다.

```text
문서 업로드
→ 텍스트 추출
→ 900자 단위 조각 생성
→ 임베딩 벡터 생성
→ ChromaDB 저장
→ 사용자 질문 입력
→ 질문 임베딩 생성
→ 본문 유사도 검색 + 파일명 검색 + 피드백 보정
→ 관련 문서와 답변 출력
```

## 전체 구동 구조도

아래 구조는 앱이 실행된 뒤 사용자가 문서를 업로드하고 질문했을 때 실제로 어떤 구성요소가 움직이는지 나타냅니다.

```text
[사용자 브라우저]
       │
       │ http://localhost:8501
       ▼
[Streamlit UI: app_streamlit.py]
       │
       ├─ 1. 문서 업로드
       │      │
       │      ▼
       │   [로컬 원본 문서 저장소]
       │   data/documents/PDF화 된 문서들/사용자 업로드/
       │      │
       │      ▼
       │   [DocumentRAG.ingest_file()]
       │      │
       │      ├─ PDF/TXT/MD/CSV 텍스트 추출
       │      ├─ 깨진 텍스트 여부 검사
       │      ├─ 900자 단위 chunk 분할
       │      ├─ 문서 제목/경로 metadata 추가
       │      └─ 임베딩 벡터 생성
       │             │
       │             ▼
       │        [ChromaDB 로컬 벡터 DB]
       │        chroma_db/
       │
       ├─ 2. 사용자 질문 입력
       │      │
       │      ▼
       │   [DocumentRAG.search()]
       │      │
       │      ├─ 질문 임베딩 생성
       │      ├─ ChromaDB 본문 유사도 검색
       │      ├─ 파일명/폴더명 토큰 검색
       │      ├─ 사용자 피드백 점수 반영
       │      └─ 참고 문서 후보 정렬
       │             │
       │             ▼
       │        [검색 결과 chunks]
       │
       ├─ 3. 답변 생성
       │      │
       │      ▼
       │   [Ollama 로컬 LLM]
       │   qwen2.5:7b-instruct
       │      │
       │      └─ 검색 결과 chunk를 근거로 답변 생성
       │
       ├─ 4. 다운로드 처리
       │      │
       │      ├─ 민감정보 없음 → 원본 다운로드
       │      └─ 민감정보 감지 → 마스킹본 다운로드
       │
       ├─ 5. 피드백 저장
       │      │
       │      ▼
       │   data/feedback/search_feedback.jsonl
       │
       └─ 6. 대화 기록 저장
              │
              ▼
          chat_histories/
```

| 구성요소 | 역할 | 저장 위치 |

### 핵심 포인트

- `localhost:8501`은 현재 PC에서 실행 중인 Streamlit 앱 주소입니다.
- Ollama와 ChromaDB는 모두 로컬에서 동작합니다.
- 외부 DB 서버는 사용하지 않습니다.
- GitHub 저장소에는 기본 문서, 원본 데이터, 벡터 DB, 대화 기록이 포함되지 않습니다.
- 새 문서를 업로드하면 원본은 `data/documents/...`에 저장되고, 검색용 벡터는 `chroma_db/`에 저장됩니다.

## RAG 검색 점수 계산 방식

검색은 한 가지 기준만 사용하지 않고, 아래 세 가지 신호를 합쳐서 참고 문서 순위를 정합니다.

```text
본문 의미 유사도 + 일반 파일명/폴더명 점수 + 제한된 사용자 피드백 점수 = 최종 검색 순위
```

### 1. 본문 의미 유사도

업로드한 문서는 먼저 텍스트로 추출됩니다. 추출된 텍스트는 약 900자 단위 조각으로 나뉘고, 각 조각은 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` 임베딩 모델을 통해 숫자 벡터로 변환됩니다.

사용자가 질문을 입력하면 질문도 같은 방식으로 벡터로 변환됩니다. ChromaDB는 질문 벡터와 문서 조각 벡터 사이의 거리를 계산해서 의미가 가까운 문서 조각을 찾습니다.

앱에서는 ChromaDB의 `distance` 값을 받아서 정렬에 사용합니다. 거리가 낮을수록 질문과 문서 조각이 더 가깝다고 판단합니다.

### 2. 일반 파일명/폴더명 점수

본문 검색만으로는 파일명이나 양식명이 중요한 질문을 놓칠 수 있습니다. 그래서 앱은 문서 본문 검색과 별도로 파일명/폴더명도 직접 검사합니다.

질문에서 의미 있는 토큰을 뽑은 뒤 파일명과 경로에 포함되어 있는지 확인합니다.

```text
토큰이 파일명/경로에 포함됨: +3
토큰이 파일명에 직접 포함됨: +3
토큰이 상위 폴더명에 포함됨: +2
질문에서 요청한 파일 형식과 실제 확장자가 맞음: +2
파일명/폴더명 점수 총합 상한: 최대 +10
```

초기 버전에는 `동아리`, `예산`, `물품`, `재료비`처럼 특정 업무 데이터에 맞춘 하드코딩 보너스가 있었습니다.

```text
예전 방식:
동아리 +5
예산 +4
물품 +4
엑셀 관련 업무 단어 +5
```

이 방식은 기존 데이터에는 잘 맞지만, 새 문서가 들어왔을 때 특정 키워드가 들어간 문서가 과하게 위로 올라오는 문제가 있었습니다. 그래서 현재 버전에서는 특정 업무 키워드 보너스를 제거하고, 일반 토큰 매칭과 확장자 매칭만 사용합니다.

파일명/폴더명 점수는 검색 결과의 `distance` 값으로 변환되어 본문 검색 결과와 함께 정렬됩니다. 점수가 높을수록 더 앞에 나올 가능성이 커지지만, 최대 +10으로 제한되어 본문 의미 유사도를 과도하게 누르지 않도록 했습니다.

### 3. 사용자 피드백 점수

검색 결과에서 `이 문서가 맞음` 또는 `이 문서 아님`을 누르면 앱은 피드백을 `data/feedback/search_feedback.jsonl`에 저장합니다.

이 피드백은 임베딩 모델을 다시 학습시키는 것은 아닙니다. 대신 다음 검색에서 같은 질문 또는 비슷한 키워드의 질문이 들어오면 검색 순위에 가산점/감점을 줍니다.

초기 버전에서는 피드백을 누를 때마다 로그가 계속 쌓이는 방식이었습니다. 이 방식은 같은 문서를 여러 번 누르면 점수가 계속 커져 특정 문서가 고정적으로 상위에 뜨는 문제가 생길 수 있었습니다.

현재 버전은 `query_key + source_path` 조합별로 최신 피드백 하나만 유지합니다.

```text
같은 질문-문서 조합에서 이 문서가 맞음 클릭: positive 상태로 덮어쓰기
같은 질문-문서 조합에서 이 문서 아님 클릭: negative 상태로 덮어쓰기
같은 조합을 여러 번 눌러도 점수는 누적되지 않음
```

피드백 점수도 상한을 둡니다.

```text
이전 질문과 현재 질문의 키워드가 완전히 같음: ±6
이전 질문과 현재 질문의 키워드가 일부 겹침: ±(4 × 키워드 겹침 비율)
키워드 겹침 비율이 0.35보다 낮음: 피드백 반영 안 함
피드백 점수 총합 상한: 최대 +6 / 최소 -6
```

예를 들어 `동아리 예산 엑셀`로 검색했을 때 어떤 문서에 `이 문서가 맞음`을 누르면, 다음에 같은 질문을 했을 때 해당 문서는 `+6`점을 받아 더 위로 올라옵니다. 반대로 `이 문서 아님`을 누르면 `-6`점이 적용되어 순위가 내려갑니다.

비슷한 질문의 경우에는 키워드가 얼마나 겹치는지에 따라 가산점이 작게 들어갑니다.

```text
이전 질문: 동아리 예산 엑셀
현재 질문: 동아리 물품 예산
겹치는 키워드: 동아리, 예산
피드백 점수: 4 × 겹침 비율
```

### 4. 최종 정렬

앱은 파일명 검색 결과와 본문 유사도 검색 결과를 합친 뒤 중복을 제거합니다. 이후 아래 기준으로 정렬합니다.

```text
1순위: distance가 낮은 결과
2순위: 파일명/폴더명 rank_score가 높은 결과
3순위: source_path 문자열 순서
```

피드백 점수가 있으면 `distance`를 낮추거나 `rank_score`를 높여서 결과가 더 앞에 나오도록 조정합니다.

정리하면, `이 문서가 맞음` 버튼은 문서 내용을 새로 학습시키는 기능이 아니라 검색 랭킹을 사용자의 선택에 맞게 보정하는 기능입니다. 다만 현재 버전에서는 피드백이 무한 누적되지 않도록 최신 상태만 유지하고, 점수 상한을 둡니다.

## 검색 개선 과정에서 해결한 문제

초기 검색 방식은 특정 문서 묶음에 잘 맞도록 업무 키워드 보너스를 크게 부여했습니다. 이 방식은 기존 데이터에서는 빠르게 원하는 파일을 찾는 장점이 있었지만, 새 데이터가 들어오면 다음 문제가 생길 수 있었습니다.

- 특정 키워드가 들어간 문서가 실제 내용보다 과하게 우선됨
- `이 문서가 맞음`을 여러 번 누르면 피드백 점수가 계속 누적됨
- 시간이 지날수록 일부 문서가 항상 상위에 고정될 수 있음

현재 버전에서는 아래 방식으로 수정했습니다.

- 특정 업무 키워드 보너스 제거
- 파일명/폴더명 점수 상한을 최대 +10으로 제한
- 피드백 점수 상한을 최대 +6 / 최소 -6으로 제한
- 같은 질문-문서 조합의 피드백은 최신 선택 하나만 유지
- 새 문서가 들어와도 일반 토큰 매칭과 본문 유사도를 중심으로 검색

## 최근 업데이트

- 참고 문서 다운로드 버튼을 렌더링할 때 일부 `.xlsx` 파일의 손상된 스타일 정보 때문에 `openpyxl`에서 `IndexError: list index out of range`가 발생하던 문제를 방어 처리했습니다.
- 민감정보가 감지된 Office 파일의 마스킹본 생성에 실패하면 앱 전체가 중단되지 않고, 해당 파일 다운로드만 제한 안내를 표시합니다.

## 데이터 폴더

앱 실행 중 자동으로 아래 폴더가 사용됩니다.

```text
data/documents/PDF화 된 문서들/사용자 업로드/
data/documents/원본데이터/사용자 업로드/
chroma_db/
chat_histories/
data/feedback/
```

각 폴더의 역할은 다음과 같습니다.

- `data/documents/원본데이터/...`: 업로드한 원본 문서
- `data/documents/PDF화 된 문서들/...`: 검색 학습용 문서
- `chroma_db/`: 문서 검색용 벡터 DB
- `chat_histories/`: 대화 기록
- `data/feedback/`: 검색 결과 피드백

이 폴더들은 개인 데이터가 포함될 수 있으므로 GitHub에 올리지 않습니다.
